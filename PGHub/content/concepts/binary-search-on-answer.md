## intro
Binary search isn't only for finding an element inside a sorted array. When the answer to a problem is a number with a monotone feasibility property — "if X works, every X' > X also works" — you can binary-search the answer space directly. Classic instances: "split array into k subarrays minimizing the largest sum", "ship packages within D days", "minimum eating speed for Koko". The shape is always the same: pick a candidate answer, run a linear-time feasibility check, narrow the range.

## whyItMatters
A brute-force "try every possible answer" sweeps a range that is often 10^9 wide. Binary search collapses that to about 30 iterations, each one a cheap O(n) scan. The total cost drops from astronomical to O(n log V) where V is the value range. The technique unlocks an entire family of LeetCode hard problems that look unrelated on the surface but share the exact same skeleton.

## intuition
The technique exists because many optimisation problems have a brute-force "try every possible answer" structure with an answer range of 10⁹ or larger — enumerating every integer is hopelessly slow. The escape route is to reframe the optimisation as a *decision* problem with a monotone feasibility predicate, then binary-search the smallest (or largest) value for which the predicate holds.

The decisive observation: most "minimum/maximum X subject to constraint Y" problems have monotone feasibility. "What is the minimum ship capacity to deliver in D days?" can be rewritten as `can_do(capacity)` — and crucially, if `can_do(15)` is true, then `can_do(16)`, `can_do(17)`, and every larger capacity must also be true (you can finish in D days with extra slack). That monotonicity is the same property that makes vanilla binary search valid; it just lives over the *answer axis* (the value space) instead of an array index space. You are still finding the leftmost True in a sorted False-False-…-True-True-True sequence — the only difference is that the sequence is implicit, generated by calling `can_do(mid)`.

The recipe is two steps:
1. Identify the monotone feasibility predicate `canDo(x)` — a single boolean function of x that is True iff x is a feasible answer. It must be monotone: if `canDo(x)` is True, so is `canDo(x')` for all x' on one side of x (depending on whether the problem minimises or maximises).
2. Binary-search the answer space [lo, hi]. Standard template: `while lo < hi: mid = (lo + hi) // 2; if canDo(mid): hi = mid else: lo = mid + 1`. When `lo == hi`, that's the optimal answer.

The reframing trick — "is X feasible?" instead of "what is the answer?" — unlocks an entire family of LeetCode hard problems that look unrelated on the surface but share the exact same skeleton: capacity to ship within D days (LC 1011), split array largest sum (LC 410), Koko eating bananas (LC 875), aggressive cows (Codeforces), magnetic force between balls (LC 1552), minimum-time to repair cars (LC 2594). The complexity drops from O(V · n) (enumerate every answer × per-check cost) to O(log V · n) — typically 30× n operations, fast enough for any 1-second time limit.

Trigger phrases for recognition: "minimum X such that...", "largest Y subject to...", "minimise the maximum", "maximise the minimum". When you see one of those, the first technique to try is binary search on the answer.

## bruteForce
Enumerate every candidate answer from the natural lower bound to the upper bound and call the feasibility check on each. For the "split array" problem, that means trying every integer from max(arr) to sum(arr) — up to 10^9 calls of an O(n) check. Quadratic-to-linear-in-the-range, and time-limit-exceeded on any non-trivial input.

## optimal
**Technique: binary search the answer space via monotone feasibility predicate.** O(n log V) where n is the feasibility check cost and V is the answer-range size. Optimal up to log factors: any algorithm must verify a candidate answer takes O(n) (you must examine every input), and binary search across the value space saturates the lower bound — Ω(log V) checks are required to distinguish between V possible answers when the only oracle is a feasibility test.

```python
def split_array_min_largest_sum(nums, k):
    def can_split(limit):                    # feasibility predicate
        parts, current = 1, 0
        for x in nums:
            if current + x > limit:           # start a new part
                parts += 1
                current = x
            else:
                current += x
        return parts <= k

    lo, hi = max(nums), sum(nums)             # tight bounds
    while lo < hi:
        mid = (lo + hi) // 2
        if can_split(mid):                    # try a smaller limit
            hi = mid
        else:                                  # need a larger limit
            lo = mid + 1
    return lo
```

Key lines: `lo, hi = max(nums), sum(nums)` defines the tightest valid bracket — the answer can never be less than the largest element (one part holds that element) and never more than the sum of all elements (one part holds everything). `can_split(mid)` is the feasibility predicate; the greedy left-to-right packing inside is the optimal way to test "can we fit into k parts with max part-sum ≤ limit". `if can_split(mid): hi = mid else: lo = mid + 1` is the canonical "leftmost-true" search — when feasible, the answer might be smaller, so move hi down; when infeasible, the answer must be larger, so move lo up. The loop terminates when `lo == hi`, which is the smallest feasible limit.

The template covers an entire family of problems. The feasibility predicate changes but the binary-search skeleton is identical:
- **Ship within D days** (LC 1011): `can_ship(cap)` = simulate days needed at given capacity.
- **Koko bananas** (LC 875): `can_finish(speed)` = sum of ceil(pile / speed) ≤ H.
- **Aggressive cows**: `can_place(min_dist)` = greedy place k cows with at least min_dist apart.
- **Painter partition** (LC 410 variant): same predicate as split-array.

**Why not enumerate every answer?** Linear scan of the value range V is O(V·n) — for V = 10⁹ and n = 10⁵ that's 10¹⁴ ops, hopeless. Binary search cuts to ~30 iterations × n = 3·10⁶ ops, instantaneous. **Why not ternary search?** Ternary search applies to unimodal functions (single peak/valley); binary search applies to monotone predicates. The two are different tools for different problem shapes. **Why monotonicity matters**: if the feasibility predicate is non-monotone, binary search is silently wrong — it can converge to the wrong answer. Always sanity-check monotonicity with hand examples before trusting the template. **Common bugs**: wrong initial bounds (lo too high or hi too low silently misses the answer); off-by-one in the move rule (mixing `hi = mid - 1` with `lo = mid + 1` causes infinite loops); returning `mid` instead of `lo` at the end.

## complexity
time: O(n log V) where n is the input size used inside the feasibility check and V is the size of the answer range.
space: O(1) auxiliary beyond what the feasibility check needs.
notes: Always confirm both endpoints of [lo, hi] are valid — a too-tight upper bound will silently miss the answer, and a too-loose one wastes only one extra iteration.

## pitfalls
- Off-by-one in the binary-search loop — prefer the `while lo < hi: mid = (lo+hi)//2; ... hi = mid or lo = mid+1` template; mixing the two move rules causes infinite loops.
- Wrong initial bounds — for "minimum capacity to ship", lo must be at least max(weight), or you will declare an infeasible capacity feasible.
- Feasibility check that isn't actually monotone — sanity-check with a few hand examples before trusting binary search.
- Returning `mid` instead of `lo` at the end; `lo` is the converged answer.

## interviewTips
- Recognize the trigger phrases: "minimum X such that ...", "largest Y subject to ...", "minimize the maximum", "maximize the minimum".
- Verbalize the predicate first: "if I can do it with capacity C, I can do it with any capacity > C." That sentence is the whole technique.
- Mention complexity in terms of the value range, not the array size — interviewers care that you understand log V.

```javascript
function splitArrayMinLargestSum(nums, k) {
  const canSplit = (limit) => {
    let parts = 1, current = 0;
    for (const x of nums) {
      if (current + x > limit) { parts++; current = x; }
      else current += x;
    }
    return parts <= k;
  };

  let lo = Math.max(...nums);
  let hi = nums.reduce((a, b) => a + b, 0);
  while (lo < hi) {
    const mid = (lo + hi) >> 1;
    if (canSplit(mid)) hi = mid;
    else lo = mid + 1;
  }
  return lo;
}
```

```java
class Solution {
    public int splitArray(int[] nums, int k) {
        int lo = 0, hi = 0;
        for (int x : nums) { lo = Math.max(lo, x); hi += x; }
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (canSplit(nums, k, mid)) hi = mid;
            else lo = mid + 1;
        }
        return lo;
    }
    private boolean canSplit(int[] nums, int k, int limit) {
        int parts = 1, current = 0;
        for (int x : nums) {
            if (current + x > limit) { parts++; current = x; }
            else current += x;
        }
        return parts <= k;
    }
}
```
