## intro
A permutation of a sequence is any arrangement of its elements in a particular order. For n distinct elements there are exactly n! permutations, and the canonical way to generate them is depth-first search with backtracking: at every level pick an unused element, recurse, then undo the pick.

## whyItMatters
- Constraint solvers — Google OR-Tools' CP-SAT, Microsoft's Z3, MiniZinc — all use backtracking with choose/recurse/unchoose at their core; permutation enumeration is the simplest non-trivial instance.
- Database join-order enumeration in PostgreSQL's `geqo` planner and DuckDB's optimizer DPccp enumerates valid join trees with the same template.
- Test-case fuzzers (American Fuzzy Lop, libFuzzer) generate input orderings via systematic backtracking when exhaustive coverage is required.
- Genetic algorithm initialization in routing solvers (Vehicle Routing Problem in OR-Tools) starts from randomly permuted seeds generated by exactly this enumeration.
- Permutations are the gateway problem for the entire backtracking family — N-Queens, sudoku, word search, combinations, subset-sum — all use the same skeleton with different constraints.

## intuition
The brute-force "generate every length-n index string then filter for no-repeats" wastes work proportional to n^n / n!, which for n = 8 is about 1600x slower than necessary. Backtracking earns its name by refusing to generate invalid prefixes in the first place. Picture a decision tree: the root has n branches for the first element, each child has n-1 branches for the second, and so on down to leaves at depth n where each leaf is one of the n! permutations. DFS visits each leaf exactly once. The only state required is "which elements have already been used in this branch" — a boolean array of size n. Before recursing into a child, mark the chosen element as used; after the recursive call returns, unmark it. This choose/recurse/unchoose discipline keeps the `used` array in lockstep with the current DFS path, so siblings of the chosen branch see a clean state with only the path-prefix elements marked. The recursion terminates naturally when the path length equals n — that node is a leaf and we record a copy. The deep insight is that backtracking encodes constraints implicitly through the choose-recurse-unchoose protocol: any predicate you can check incrementally (no-repeats here, queen-attack avoidance in N-Queens, region-validity in sudoku) prunes whole subtrees before they explode. The in-place-swap variant achieves the same effect by physically moving the chosen element to position k and recursing on `[k+1..n-1]`; it avoids the `used` array entirely at the cost of slightly less obvious control flow.

## visualization
For [1,2,3]: pick 1 → pick 2 → pick 3 → leaf [1,2,3], backtrack, pick 3 → pick 2 → leaf [1,3,2], backtrack twice, pick 2 → pick 1 → pick 3 → leaf [2,1,3], and so on. The recursion tree has 3! = 6 leaves; each path from root to leaf is one permutation. The "used" array flips a bit on entry and unflips on exit, exactly tracking the current path.

## bruteForce
Generate every length-n string of indices in [0, n), then keep only those with no repeats. That is n^n candidates filtered down to n! — wasteful by a factor of n^n / n!, which for n=8 is about 1600x extra work. It also requires post-filtering, which obscures the underlying combinatorial structure.

## optimal
Depth-first search with a used-array (or in-place swap variant). Maintain a current partial permutation `path` and a boolean array `used` of size n. At each call, iterate i from 0 to n-1; if `used[i]` is false, mark it true, push `nums[i]` onto path, recurse, then pop path and unmark `used[i]`. When `path` reaches length n, record `path[:]` (copy, not reference) into the result. The work is proportional to the recursion tree size, which is bounded by n * n! — n! leaves plus internal nodes, with O(n) per leaf to copy the path. This is asymptotically optimal because the output itself contains n * n! integers, so any algorithm must do that much writing.

```python
def permute(nums):
    res, path, used = [], [], [False] * len(nums)

    def dfs():
        if len(path) == len(nums):
            res.append(path[:])         # critical: copy, not reference
            return
        for i, x in enumerate(nums):
            if used[i]:                 # skip already-chosen elements
                continue
            used[i] = True
            path.append(x)
            dfs()                       # recurse with the chosen element marked
            path.pop()                  # unchoose: restore state for siblings
            used[i] = False
    dfs()
    return res
```

The `path[:]` copy is the most common bug source — appending the live reference makes every result entry alias the (eventually empty) `path`. The choose/recurse/unchoose protocol is the discipline that keeps `used` consistent with `path` at every level. For duplicate inputs (e.g., `[1, 1, 2]`), sort the array first and skip when `i > 0 and nums[i] == nums[i-1] and not used[i-1]` to deduplicate without re-checking — same skeleton, one extra guard.

## complexity
time: O(n * n!)
space: O(n) auxiliary plus O(n * n!) for the output if collected
notes: There are n! leaves and each leaf does O(n) work to copy the path into the output. The recursion stack is O(n) deep. If duplicates are allowed, sort first and skip `i > 0 && nums[i] == nums[i-1] && !used[i-1]` to de-duplicate without re-checking.

## pitfalls
- Appending the live `path` reference into results — every entry ends up pointing at the same (now-empty) list. Always copy.
- Forgetting to unmark `used[i]` on the way back up — the next branch sees stale state.
- Not handling duplicates: [1,1,2] produces 6 permutations naively, but only 3 are distinct.
- Recursing on n above ~10 in an interview when n! explodes — call out the runtime ceiling before coding.

## interviewTips
- Walk through n=3 on the whiteboard before writing code; it forces you to verbalize the choose / unchoose pattern.
- Mention the in-place swap variant as a follow-up — same output, O(1) extra bookkeeping beyond the call stack.
- If asked for the k-th permutation, switch to the factorial-number-system trick — never enumerate all n!.

## code.javascript
```javascript
function permute(nums) {
  const res = [], path = [], used = new Array(nums.length).fill(false);
  function dfs() {
    if (path.length === nums.length) { res.push([...path]); return; }
    for (let i = 0; i < nums.length; i++) {
      if (used[i]) continue;
      used[i] = true;
      path.push(nums[i]);
      dfs();
      path.pop();
      used[i] = false;
    }
  }
  dfs();
  return res;
}
```

## code.java
```java
public List<List<Integer>> permute(int[] nums) {
    List<List<Integer>> res = new ArrayList<>();
    boolean[] used = new boolean[nums.length];
    dfs(nums, new ArrayList<>(), used, res);
    return res;
}

private void dfs(int[] nums, List<Integer> path, boolean[] used, List<List<Integer>> res) {
    if (path.size() == nums.length) { res.add(new ArrayList<>(path)); return; }
    for (int i = 0; i < nums.length; i++) {
        if (used[i]) continue;
        used[i] = true;
        path.add(nums[i]);
        dfs(nums, path, used, res);
        path.remove(path.size() - 1);
        used[i] = false;
    }
}
```

## code.cpp
```cpp
class Solution {
public:
    vector<vector<int>> permute(vector<int>& nums) {
        vector<vector<int>> res;
        vector<int> path;
        vector<bool> used(nums.size(), false);
        dfs(nums, path, used, res);
        return res;
    }
private:
    void dfs(vector<int>& nums, vector<int>& path, vector<bool>& used, vector<vector<int>>& res) {
        if (path.size() == nums.size()) { res.push_back(path); return; }
        for (int i = 0; i < (int)nums.size(); i++) {
            if (used[i]) continue;
            used[i] = true;
            path.push_back(nums[i]);
            dfs(nums, path, used, res);
            path.pop_back();
            used[i] = false;
        }
    }
};
```
